## Context

P5R Guide 網站目前只有 Persona 數據查詢功能，玩家無法規劃合成路徑。目標是實現一個合成規劃工具，讓玩家選擇目標面具和目標技能/特性，系統計算並展示所有可能的合成路徑。

### 當前狀態
- 已有 259 個面具數據（含技能、屬性抗性、五維屬性）
- 已有合成規則文檔 `docs/FUSION_RULES.md`
- 數據存儲在 SQLite + JSON

### 約束
- 前端：Next.js 16 + TypeScript + Tailwind CSS
- 需在客戶端計算（無後端服務器）
- 性能要求：路徑計算需在 3 秒內完成

## Goals / Non-Goals

**Goals:**
- 實現基於動態規劃的合成路徑搜尋引擎
- 支援按目標面具 + 目標技能/特性過濾
- 展示多條合成路徑及代價
- 響應式設計，支援移動端

**Non-Goals:**
- 不實現多體合成（只做二體合成）
- 不實現合成事故預測
- 不實現網絡合成
- 不實現覺醒路徑規劃

## Decisions

### 1. 算法選擇：動態規劃 vs 暴力枚舉

**決定：動態規劃 + BFS**

理由：
- 暴力枚舉複雜度為 O(n!)，無法接受
- 動態規劃預計算所有可能的結果，搜尋時只需查表
- BFS 可找到最短路徑（最少合成次數）

**替代方案考慮：**
- A* 啟發式搜尋 - 不適用於此場景（無法定義有意義的啟發函數）
- 貪心演算法 - 可能錯過最優解

### 2. 數據結構：矩陣 vs 映射

**決定：使用 Map 結構存儲合成表**

理由：
- 22x22 矩陣為 484 個条目，使用 Map 可用 `{arcanaA}:{arcanaB}` 作為 key
- 查詢複雜度為 O(1)
- 更易於擴展（例如支援多體合成）

### 3. 路徑展示策略

**決定：顯示前 10 條最優路徑**

理由：
- 顯示所有路徑可能數量龐大（指数级）
- 使用代價（合成價格）排序，優先展示低代價路徑
- 用戶可展開查看詳細資訊

### 4. 技能繼承處理

**決定：從結果向前推導**

理由：
- 根據目標技能，反向查找哪些面具可以繼承該技能
- 遞歸處理：找到能繼承目標技能的面具 A，再找能合成 A 的面具 B
- 需預先建立「技能->面具」的反向索引

## Risks / Trade-offs

### [風險] 路徑數量爆炸
- **描述**：某些目標可能產生數百條路徑
- **緩解**：限制顯示數量，按代價排序只顯示前 10 條

### [風險] 循環依賴
- **描述**：可能出現 A->B->A 的循環
- **緩解**：BFS 搜尋時記錄已訪問節點，避免重複

### [風險] 性能瓶頸
- **描述**：大規模路徑計算可能超過 3 秒
- **緩解**：
  - 預計算合成結果矩陣
  - 使用 memoization 快取中間結果
  - 限定最大搜尋深度為 5 層

### [權衡] 準確性 vs 速度
- **描述**：完整的最優解需要考慮所有技能繼承組合
- **緩解**：優先確保找到有效路徑，再優化排序

## Migration Plan

1. **Phase 1: 數據準備**
   - 將合成矩陣導入程式碼
   - 建立技能->面具反向索引
   - 建立寶魔加成表

2. **Phase 2: 核心演算法**
   - 實現 `calculateFusionResult(arcanaA, arcanaB)` 函數
   - 實現 `findFusionPaths(target, constraints, maxDepth)` 函數
   - 實現 `calculateFusionPrice(path)` 函數

3. **Phase 3: UI 實現**
   - 創建 FusionPlanner 組件
   - 實現目標選擇器（面具下拉選單）
   - 實現技能/特性過濾器
   - 實現路徑卡片展示

4. **Phase 4: 整合測試**
   - 端到端測試各種合成場景
   - 性能測試優化
